# Implementation Plan: `hadron spec check-tools` (tool-name drift lint)

> **Status: implemented and verified** on this branch; reflects the design as
> built. Resolves **part 3** of
> [#240](https://github.com/hadron-memory/hadron-cli/issues/240) — the check that
> prevents `hadron_*` tool references in specs from silently rotting (the root
> cause of the corpus-wide rename that motivated `spec grep`/`spec replace`).

## Context

The stale `h-*` shorthand drifted because nothing tied the tool names written in
specs to the real registered tools. This adds that anchor: a check that flags any
`hadron_*` token in a spec that is not a registered tool.

Dogfooding `spec grep 'hadron_[a-z_]+' --regex` over the live corpus surfaced two
realities the issue's "extraction is a one-liner" assumption missed:

1. **Two registries.** A spec legitimately cites tools from *both* hadron-server
   registries — MCP tools (`server.tool('hadron_*')` in `src/mcp/server.ts`) **and**
   runner tools (`RunToolDef { name: 'hadron_*' }` in `src/lib/runner/tools/*.ts`,
   e.g. `hadron_read_secret`, `hadron_store_file`). A manifest of only the MCP
   registry would false-positive on the runner tools. The manifest is the **union**
   (47 tools).
2. **Genuine non-tools.** `hadron_token` (the web sign-in cookie) and `hadron_chatbot`
   (the `hadron_chatbot_*` tool-family prefix, written as a glob) are `hadron_*`
   identifiers that legitimately appear in prose but aren't tools.

## Design (agreed with the owner)

- **A dedicated `spec check-tools` command**, not a rule inside `spec lint` — so the
  generic, any-corpus `spec lint` stays free of a Hadron-specific baked-in tool list.
- **Manifest source**: `internal/cmd/spec/mcp-tools.txt`, generated by
  `scripts/gen-tools-manifest.sh` (union of both registries), regenerated with
  `make tools-manifest`, embedded via `go:embed`. **CI drift-check** (`make
  tools-manifest-check`) runs in the nightly `schema-drift` workflow and fails if the
  committed manifest lags hadron-server — mirroring the schema-snapshot pattern.
- **Ignore-list**: `internal/cmd/spec/mcp-tools-ignore.txt`, hand-maintained (the two
  known non-tools), also embedded. Kept minimal — a genuinely-unregistered reference
  should be fixed in the spec, not silenced.

## Command

```
hadron spec check-tools [-m <memory>] [--prefix <loc>] [--json]
```

- Reuses `spec grep`'s spec-scoped bulk read: list the spec-tagged citation nodes,
  read their content + abstract in one `nodeBatch`, extract `hadron_*` tokens
  client-side.
- Token regex `hadron_[a-z]+(?:_[a-z]+)*` — underscore-joined letter segments, so a
  trailing `_` in a `hadron_chatbot_*` glob isn't captured and the token compares
  cleanly to a registered name.
- Flags any token that is neither registered nor ignored, as
  `citation:field:line: token`. **Exits 5** (conflict) on findings — like `spec
  lint`, so CI can gate on it; a clean corpus prints a one-line OK and exits 0.
- `--json` emits `[{citation, field, line, token, text}]`.

## Files

- `scripts/gen-tools-manifest.sh` — the union extractor (emits a fixed header so the
  committed file is byte-reproducible for the drift-check).
- `internal/cmd/spec/mcp-tools.txt` — generated manifest (47 tools).
- `internal/cmd/spec/mcp-tools-ignore.txt` — hand-curated non-tools.
- `internal/cmd/spec/checktools.go` — the command + `go:embed` + parser/extractor.
- `Makefile` — `tools-manifest` (regenerate) and `tools-manifest-check` (drift).
- `.github/workflows/schema-drift.yml` — second drift axis on the existing gated
  server checkout; files its own de-duped tracking issue.

## Tests

- Unit (`checktools_test.go`): `parseToolList` (whole-line + inline `#` comments),
  `checkToolField` (registered pass, unknown flagged, ignored skipped, per-line
  dedup), the token regex's no-trailing-underscore behavior, and a guard that the
  embedded manifest/ignore-list are present and sane (a broken `go:embed` would
  silently disable the check).
- Command-level (`spec_commands_test.go`): a bogus `hadron_bogus_tool` is flagged
  (exit 5, `--json`) while a registered `hadron_get_node` and the ignored
  `hadron_token` pass (exit 0).

## Verification

Dogfooded end-to-end against the live `hadronmemory.com::specs` corpus: `spec
check-tools` reports **clean** against the 47-tool manifest — the whole pipeline
(bulk read → extract → manifest + ignore-list) validated on real data. `make
tools-manifest-check` passes against a fresh server `main`.
